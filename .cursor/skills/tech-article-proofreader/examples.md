## 出力例（指摘一覧）

### 例1: 表記ゆれ＋誤字

#### 表記統一候補

- 「TypeScript」表記に統一（Typescript → TypeScript）
- 「GitHub Actions」表記に統一（Github Actions → GitHub Actions）

- **[Minor] 表記ゆれ**
  - **位置**: 「## セットアップ」付近 / `Typescript をインストールします`
  - **指摘**: 製品名の表記が揺れています
  - **修正案**: `Typescript` → `TypeScript`
  - **理由**: 固有名詞は公式表記に合わせると検索性と信頼性が上がります

- **[Minor] 誤字**
  - **位置**: 「## まとめ」付近 / `解説したいます`
  - **指摘**: 変換ミス（誤字）です
  - **修正案**: `解説したいます` → `解説したいです`
  - **理由**: 読み手の引っ掛かりを減らします

### 例2: 技術的に不確かな断定（要確認付き）

- **[Major] 技術正確性**
  - **位置**: 「## パフォーマンス」付近 / `この設定で必ず高速化します`
  - **指摘**: 環境やケースによっては効果が出ない可能性があります（断定が強い）
  - **修正案**:
    - Before: `この設定で必ず高速化します`
    - After: `ケースによっては高速化が期待できます（要計測）。まずは計測して効果を確認してください`
  - **理由**: 読者の誤解を避け、再現性/検証可能性を担保できます
  - **備考**: 効果が出る条件（ボトルネック箇所、計測方法）を1〜2行で補足できるとより良いです
